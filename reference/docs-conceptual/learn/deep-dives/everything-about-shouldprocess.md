<span data-ttu-id="65fc3-249">Wenn der Benutzer `-Force` angibt, möchten wir die Bestätigungsaufforderung unterdrücken, es sei denn, es wird auch `-Confirm` angegeben.</span><span class="sxs-lookup"><span data-stu-id="65fc3-249">If the user specifies `-Force`, we want to suppress the confirm prompt unless they also specify `-Confirm`.</span></span> <span data-ttu-id="65fc3-250">Dadurch kann ein Benutzer eine Änderung erzwingen, diese aber dennoch bestätigen.</span><span class="sxs-lookup"><span data-stu-id="65fc3-250">This allows a user to force a change but still confirm the change.</span></span> <span data-ttu-id="65fc3-251">Dann legen wir `$ConfirmPreference` im lokalen Geltungsbereich fest.</span><span class="sxs-lookup"><span data-stu-id="65fc3-251">Then we set `$ConfirmPreference` in the local scope.</span></span> <span data-ttu-id="65fc3-252">Durch Verwenden des Parameters `-Force` wird der Parameter `$ConfirmPreference` vorübergehend auf „none“ festgelegt, wodurch die Bestätigungsaufforderung deaktiviert wird.</span><span class="sxs-lookup"><span data-stu-id="65fc3-252">Now, using the `-Force` parameter temporarily sets the `$ConfirmPreference` to none, disabling prompt for confirmation.</span></span>

```powershell
if ($Force -or $PSCmdlet.ShouldProcess('TARGET')){
        Write-Output "Some Action"
    }
```

<span data-ttu-id="65fc3-253">Wenn sowohl `-Force` als auch `-WhatIf` angegeben werden, muss `-WhatIf` Priorität haben.</span><span class="sxs-lookup"><span data-stu-id="65fc3-253">If someone specifies both `-Force` and `-WhatIf`, then `-WhatIf` needs to take priority.</span></span> <span data-ttu-id="65fc3-254">Dieser Ansatz behält die `-WhatIf`-Verarbeitung bei, weil `ShouldProcess` immer ausgeführt wird.</span><span class="sxs-lookup"><span data-stu-id="65fc3-254">This approach preserves `-WhatIf` processing because `ShouldProcess` always gets executed.</span></span>

### <a name="shouldcontinue--force"></a><span data-ttu-id="65fc3-257">ShouldContinue: -Force</span><span class="sxs-lookup"><span data-stu-id="65fc3-257">ShouldContinue -Force</span></span>

<span data-ttu-id="65fc3-258">Im Folgenden sehen Sie die richtige Vorgehensweise zum Implementieren von `-Force` mit `ShouldContinue`.</span><span class="sxs-lookup"><span data-stu-id="65fc3-258">This is the correct way to implement `-Force` with `ShouldContinue`.</span></span>

```powershell
function Test-ShouldContinue {
    [CmdletBinding()]
    param(
        [Switch]$Force
    )

    if($Force -or $PSCmdlet.ShouldContinue('TARGET','OPERATION')){

    }

```

<span data-ttu-id="65fc3-259">Durch Platzierung links vom `-or`-Operator wird `$Force` zuerst ausgewertet.</span><span class="sxs-lookup"><span data-stu-id="65fc3-259">By placing the `$Force` to the left of the `-or` operator, it gets evaluated first.</span></span> <span data-ttu-id="65fc3-260">Auf diese Weise wird die Ausführung der `if`-Anweisung umgangen.</span><span class="sxs-lookup"><span data-stu-id="65fc3-260">Writing it this way short circuits the execution of the `if` statement.</span></span> <span data-ttu-id="65fc3-261">Wenn `$force` den Wert `$true` aufweist, wird `ShouldContinue` nicht ausgeführt.</span><span class="sxs-lookup"><span data-stu-id="65fc3-261">If `$force` is `$true`, then the `ShouldContinue` is not executed.</span></span>

<span data-ttu-id="65fc3-262">Wir müssen uns in diesem Szenario nicht um `-Confirm` oder `-WhatIf` kümmern, da sie von `ShouldContinue` nicht unterstützt werden.</span><span class="sxs-lookup"><span data-stu-id="65fc3-262">We don't have to worry about `-Confirm` or `-WhatIf` in this scenario because they're not supported by `ShouldContinue`.</span></span> <span data-ttu-id="65fc3-263">Aus diesem Grund muss die Verarbeitung anders erfolgen als bei `ShouldProcess`.</span><span class="sxs-lookup"><span data-stu-id="65fc3-263">This is why it needs to be handled differently than `ShouldProcess`.</span></span>

## <a name="scope-issues"></a><span data-ttu-id="65fc3-264">Probleme mit dem Gültigkeitsbereich</span><span class="sxs-lookup"><span data-stu-id="65fc3-264">Scope issues</span></span>

<span data-ttu-id="65fc3-265">Die Verwendung `-WhatIf` und `-Confirm` sollte für alle Aspekte in Ihren Funktionen sowie für alle Elemente gelten, die von Ihren Funktionen aufgerufen werden.</span><span class="sxs-lookup"><span data-stu-id="65fc3-265">Using `-WhatIf` and `-Confirm` are supposed to apply to everything inside your functions and everything they call.</span></span> <span data-ttu-id="65fc3-266">Dazu legen sie im lokalen Gültigkeitsbereich der Funktion `$WhatIfPreference` auf `$true` oder `$ConfirmPreference` auf `Low` fest.</span><span class="sxs-lookup"><span data-stu-id="65fc3-266">They do this by setting `$WhatIfPreference` to `$true` or setting `$ConfirmPreference` to `Low` in the local scope of the function.</span></span> <span data-ttu-id="65fc3-267">Wenn Sie eine andere Funktion aufrufen, werden diese Werte in Aufrufen von `ShouldProcess` verwendet.</span><span class="sxs-lookup"><span data-stu-id="65fc3-267">When you call another function, calls to `ShouldProcess` use those values.</span></span>

<span data-ttu-id="65fc3-271">Es gibt nur eine Situation, in der das Ganze nicht funktioniert: Wenn ein Skript oder ein Skriptmodul eine Funktion in einem anderen Skriptmodul aufruft.</span><span class="sxs-lookup"><span data-stu-id="65fc3-271">The one specific place where it doesn't work is when a script or a script module calls a function in another script module.</span></span> <span data-ttu-id="65fc3-272">Das klingt vielleicht nicht besonders problematisch, aber die meisten Module, die Sie erstellen oder aus der PSGallery abrufen, sind Skriptmodule.</span><span class="sxs-lookup"><span data-stu-id="65fc3-272">This may not sound like a big problem, but most of the modules you create or pull from the PSGallery are script modules.</span></span>
